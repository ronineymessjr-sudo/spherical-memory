## 三、QA Agent 团队 + 轮转规则

### 3.1 QA 团队编制

| QA 编号 | 角色 | 主责 | 次责 |
|---|---|---|---|
| **QA-1** | 接口守门人 | 任何 PR 必过 | 集成冒烟测试 |
| **QA-2** | 视觉与体验 | UI 一致性、动效流畅度 | 移动端可用性 |
| **QA-3** | 3D 性能 | FPS、内存、加载速度 | 2D 降级验证 |
| **QA-4** | 集成测试 | 端到端流程 | 多浏览器兼容 |
| **QA-5** | 演示就绪 | 演示场景模拟 | 应急方案验证 |

**人员配置建议**：
- 每个 QA 角色 1 人，**5 个 QA**
- **如人手不够**：QA-3 和 QA-4 可合并为同一人

### 3.2 轮转规则

**轮值表**（每个 QA 负责 review 哪几个 dev agent）：

```
QA-1（接口守门人）:  所有 PR 必过，**不轮转**
QA-2（视觉）:        A1, A11, A12, A13, A14, A22, A26
QA-3（3D 性能）:     A4, A5, A6, A7, A8, A9, A15, A16
QA-4（集成）:        A2, A3, A18, A19, A20, A21
QA-5（演示就绪）:    A23, A24, A25
```

**轮转节点**（每 4 小时调整一次）：
- Day 1 09:00：按上表分配
- Day 1 13:00：保持
- Day 1 17:00：QA-1 累的话换人顶
- Day 2 09:00：QA-2 和 QA-3 互换 2~3 个 dev agent（避免审美疲劳）
- Day 2 13:00：QA-4 接管 QA-2 部分任务（QA-2 收工）

**每个 PR 的 review 流程**：
```
Dev Agent 提 PR
   ↓
QA-1（接口守门人，10 分钟内回复）
   ├─ 接口不符 → 打回，要求修改
   └─ 通过 → 进入领域 QA review
   ↓
领域 QA（30 分钟内回复）
   ├─ 有问题 → 打回，附 checklist
   └─ 通过 → A0 合入
   ↓
A0 合并到 main
```

### 3.3 QA 工作流细则

#### QA-1 接口守门人

**检查清单**（每个 PR 必过）：
```markdown
□ 模块挂在 window.SM.modules.{category}.{name}？
□ init() 和 destroy() 都实现了？
□ 没有 import 业务模块？只 import 基础模块（bus/state/input）？
□ 没有在 init() 里 emit 事件？
□ 所有 emit 的事件都在 INTERFACES.md §7 注册了？
□ 所有方法签名与 INTERFACES.md §6.X 一致？
□ 没有在 window.SM 上加新字段（除非走 issue 流程）？
□ destroy() 释放了所有 geometry/material/texture？
```

**打回话术**：
> "接口不符：你的 `init` 在 `SM.modules.render3d.panoramaBind` 下，签名应该是 `init()` 不是 `setup()`。请对齐 INTERFACES.md §6.5。"

#### QA-2 视觉与体验

**检查清单**：
```markdown
□ 字体、颜色与封面风格一致？
□ 动画时长合理（不卡、不飘）？
□ 按钮 ≥ 44px？
□ 移动端：单手能操作？
□ 颜色对比度 ≥ 4.5:1（WCAG AA）？
□ 不依赖具体 emoji（用 SVG/icon font 兜底）？
```

**测试方法**：
- Chrome DevTools → iPhone 14 Pro 模拟
- 录屏 5s 关键动效，慢放检查
- 截图对比 A11 设定的"风格基准图"

#### QA-3 3D 性能

**检查清单**：
```markdown
□ 中端机 fps ≥ 30？（4x CPU throttle + Fast 3G）
□ 初始加载 ≤ 5s？
□ 内存 ≤ 200MB？
□ 没有 console warning（WebGL 警告也要处理）？
□ 资源（geometry/material/texture）正确释放？
□ 移动端没炸（温度上升）？
```

**测试方法**：
```js
// 性能埋点（QA-3 自己加到主控）
const perf = {
  fps: 0, memory: 0, loadTime: 0
};
SM.bus.on('sphere:ready', () => {
  perf.loadTime = Date.now() - SM.startTime;
  console.table(perf);
});
```

**性能不达标时的降级策略**（A0 决策）：
| 问题 | 降级方案 |
|---|---|
| FPS < 30 | 6 片 → 4 片 |
| 内存 > 200MB | 纹理降采样到 1024 |
| 加载 > 5s | 纹理 lazy load，按需加载 |
| 移动端炸 | 自动切 2D 兜底 |

## 五、沟通与冲突解决

### 5.1 沟通节奏

**每日站会**（**3 次，每次 10 分钟**）：
- 09:00 — 启动对齐（A0 主持）
- 13:00 — Phase 切换对齐
- 19:00 — Phase 1 收工对齐

**异步沟通**（Slack）：
- `#general` 公告
- `#dev-{phase}` 阶段讨论
- `#qa` QA 反馈
- `#blockers` 卡点求助（15 分钟无响应 @A0）

### 5.2 冲突解决优先级

```
1. INTERFACES.md（冻结后不可改）
2. 演示完整性（一切砍功能以演示能跑为准）
3. 性能（fps 优先于画质）
4. 视觉细节（动效完整度优先于像素级完美）
5. 代码质量（hackathon 不追求代码优雅，能跑能合就行）
```

### 5.3 接口冲突处理流程

```
Agent A 提 PR，要求加新事件 / 改接口
   ↓
QA-1 打回："请先开 issue"
   ↓
Agent A 开 issue，描述：
   - 为什么要加这个事件
   - 现有事件为什么不满足
   - 新事件的 payload 形状
   ↓
A0 + 2 个相关 Agent 评审（30 分钟内）
   ↓
批准 → 更新 INTERFACES.md → Agent A 重新提 PR
拒绝 → A0 解释，提供替代方案
```

### 5.4 紧急求助

**卡住超过 1 小时**：
1. 在 `#blockers` 写清楚：卡在哪、试过什么、卡了多久
2. @ 频道所有人
3. 15 分钟无响应 @ A0

**A0 失联**（如厕/吃饭）：
- A0 必须指定 1 个 backup（建议 A2，A2 最熟悉代码结构）
- 30 分钟内未回应，backup 自动接管
